# Components and States

## Purpose

Design UI components as reusable, stateful building blocks.

## Repeatable Practice

- Define a component's anatomy: label, icon, container, padding, state, helper content.
- Create variants only when they represent real usage differences.
- Define primary, secondary, tertiary, destructive, and disabled variants for actions.
- Cover default, hover, active, focus, loading, disabled, selected, and error states.
- Keep component behavior consistent across screens.

## Development Checklist

- Every interactive component has keyboard focus styling.
- Loading states prevent duplicate actions.
- Disabled states explain or imply why action is unavailable.
- Component variants are documented with intended usage.
